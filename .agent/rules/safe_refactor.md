# Beeliber Web - Safe Refactor Skill

이 가이드는 Beeliber 웹 애플리케이션의 리팩토링 요청 발생 시 준수해야 할 표준 절차와 원칙을 정의합니다.

## 리팩토링 원칙

1.  **동작 보존**: 사용자가 체감하는 UI 결과와 동작은 절대 변경하지 않습니다.
2.  **데이터 무결성**: 데이터 바인딩, API 연동 로직, 상태값(State)의 의미와 흐름을 유지합니다.
3.  **구조적 개선 집중**: 중복 코드 제거, 컴포넌트 분리, 가독성 향상, 관심사 분리(SoC)를 핵심 목표로 삼습니다.
4.  **근거 있는 공통화**: 실제 재사용이 일어나는 부분에 대해서만 공통화를 진행하며, 성급한 일반화를 경계합니다.
5.  **적정 추상화**: 코드의 복잡도를 높이는 과도한 추상화는 피하고 직관적인 구조를 유지합니다.
6.  **전후 동일성**: 리팩토링 전과 후의 비즈니스 로직 실행 결과가 100% 동일해야 합니다.

## 필수 정리 내용 (보고서 형식)

리팩토링 완료 후 다음과 같은 형식으로 보고합니다.

1.  **리팩토링 목적**: 구조 개선을 통해 얻고자 하는 이점 (유지보수성, 가독성 등)
2.  **변경 전 구조 문제**: 기존 코드에서 발견된 구조적 결함이나 개선이 필요한 지점
3.  **변경 후 구조**: 리팩토링을 통해 바뀐 코드의 구조와 설계
4.  **기능 동일성 보장 포인트**: 동작이 변하지 않았음을 검증하기 위해 확인한 주요 로직들
5.  **영향 범위**: 수정이 적용된 모듈 및 관련 컴포넌트
6.  **롤백 가능성**: 문제 발생 시 이전 상태로 복구하기 위한 가이드

## 금지 사항

*   리팩토링과 UI 변경을 동시에 진행하는 행위
*   `enum`, 데이터 스키마, API 인터페이스 등 외부 계약(Contract)을 변경하는 행위
*   충분한 테스트와 검증 없이 넓은 범위의 구조를 한 번에 변경하는 행위
