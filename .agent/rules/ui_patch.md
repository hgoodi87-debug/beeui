# Beeliber Web - UI Patch Skill

이 가이드는 Beeliber 웹 애플리케이션의 UI 수정 요청 발생 시 준수해야 할 표준 절차와 원칙을 정의합니다.

## UI 수정 원칙

1.  **시각적 요소 집중**: 레이아웃, 간격(Spacing), 텍스트 계층 구조(Hierarchy), 버튼 정렬, 컬러 및 배지(Badge) 표현 등 순수 UI 요소만 수정합니다.
2.  **기능 변경 최소화**: 명시적으로 요청된 경우에만 기능을 추가하거나 삭제합니다.
3.  **로직 보존**: API 호출, 상태 관리(State Management), 이벤트 핸들링 로직은 절대 건드리지 않습니다.
4.  **영향 범위 제어**: 공통 컴포넌트 수정보다는 로컬 화면 단에서 해결하여 다른 화면에 미치는 영향을 최소화합니다.
5.  **반응형 유지**: 기존의 반응형 레이아웃 동작을 유지합니다.
6.  **텍스트 유지**: 문구 변경 요청이 없는 한 기존 텍스트를 그대로 사용합니다.

## 필수 확인 사항

*   수정 후 데이터 렌더링이 깨지지 않는가?
*   클릭 액션 및 인터랙션이 기존과 동일하게 작동하는가?
*   상태값(Status) 표시의 의미가 기존과 일치하는가?
*   다른 화면의 스타일에 의도치 않은 영향이 없는가?

## 출력 형식 (보고서 필수 포함 내용)

UI 수정 작업 완료 후 다음과 같은 형식으로 보고합니다.

1.  **UI 수정 요약**: 어떤 변경이 이루어졌는지 간략히 설명
2.  **변경된 화면 영역**: 수정이 적용된 구체적인 컴포넌트나 섹션
3.  **변경 파일 목록**: 수정된 파일들의 절대 경로
4.  **유지된 기능/로직**: 수정 중 건드리지 않은 핵심 로직 명시
5.  **영향 범위**: 스타일 변경이 미치는 범위 (로컬/글로벌)
6.  **체크해야 할 QA 포인트**: 사용자가 중점적으로 확인해야 할 사항
