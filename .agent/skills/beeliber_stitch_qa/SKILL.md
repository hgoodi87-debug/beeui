# 🧵 Beeliber Stitch QA 프로토콜

Stitch로 UI/UX를 수정하거나 추가한 직후, **모든 직원 페르소나가 자신의 담당 영역을 체크하고 보고**합니다.

---

## ⏱️ 언제 실행하나

- Stitch에서 컴포넌트·페이지·스타일을 수정·추가한 직후
- `uipolish` 모드 작업 완료 직후
- 랜딩·예약·관리자 화면 UI 변경이 적용된 직후

---

## 👥 전 직원 페르소나 체크 분담

### 🎨 프엔이 (Frontend / Lending) — UI·연동 레이어
> 담당: 렌더링, 반응형, 컴포넌트 연결, 번역

| 체크 항목 | 확인 방법 |
|----------|---------|
| 수정된 컴포넌트가 정상 렌더링되는가 | 브라우저 화면 직접 확인 |
| 기존 클릭 이벤트·인터랙션이 그대로 작동하는가 | 버튼·링크·모달 전수 클릭 |
| 반응형 레이아웃이 깨지지 않는가 | 모바일(375px) · 태블릿(768px) · 데스크탑(1280px) |
| 번역 키가 누락되지 않았는가 | 6개 언어(`translations_split/`) 전환 확인 |
| 다른 화면에 스타일 오염이 없는가 | 수정 파일의 CSS 클래스 범위 확인 |
| 공통 컴포넌트를 건드린 경우 전 화면 영향 체크 | 연관 페이지 최소 3개 확인 |

**보고 한 줄**: `[프엔이] UI 렌더링 ✅ / 반응형 ✅ / 번역 ✅ / 오염 없음 ✅` 또는 `⚠️ [항목]: [증상]`

---

### 📊 비키 (Data / Planning) — 데이터·상태 레이어
> 담당: 상태 관리, API 연동, 데이터 렌더링 정합성

| 체크 항목 | 확인 방법 |
|----------|---------|
| UI 변경이 상태(State) 흐름을 끊지 않았는가 | 예약 플로우 처음→끝 실행 |
| API 호출 파라미터·응답 매핑이 그대로인가 | 네트워크 탭에서 요청/응답 확인 |
| 로딩·에러·빈 상태(empty state) UI가 존재하는가 | 네트워크 끊거나 빈 데이터로 테스트 |
| 수정된 컴포넌트에서 데이터가 올바르게 노출되는가 | 실제 데이터로 렌더링 확인 |
| Firestore 쿼리 결과가 UI에 정확히 반영되는가 | 관리자 화면 또는 DevTools 확인 |

**보고 한 줄**: `[비키] 상태 흐름 ✅ / API 매핑 ✅ / 빈상태 ✅` 또는 `⚠️ [항목]: [증상]`

---

### 💰 예결이 (Accounting / Backend) — 결제·보안 레이어
> 담당: 결제 흐름, 가격 표시, 권한 분기

| 체크 항목 | 확인 방법 |
|----------|---------|
| 가격 표시가 `beeliber_pricing` 기준값과 일치하는가 | 요금표 하드코딩 여부 확인 |
| 결제 버튼·플로우가 기존과 동일하게 작동하는가 | mock 결제 실행 |
| 관리자 전용 UI가 일반 사용자에게 노출되지 않는가 | 비로그인·일반 계정으로 접근 확인 |
| 환경변수(`VITE_TOSS_*`)가 UI에 노출되지 않는가 | 소스·번들 확인 |
| 민감 정보(이메일·전화번호)가 UI에 과노출되지 않는가 | 화면 직접 확인 |

**보고 한 줄**: `[예결이] 가격 표시 ✅ / 결제 흐름 ✅ / 권한 노출 없음 ✅` 또는 `⚠️ [항목]: [증상]`

---

### 🔧 스봉이 (실장 / 종합 검토) — 빌드·SEO·브랜드 레이어
> 담당: 빌드 통과, 브랜드 금지어, SEO 메타태그, 최종 승인

| 체크 항목 | 확인 방법 |
|----------|---------|
| `npm run build` 통과하는가 | 빌드 실행 후 오류 없음 확인 |
| 브랜드 금지어가 추가된 텍스트에 없는가 | `beeliber_master` 금지어 목록 대조 |
| 신규 화면의 SEO 메타태그가 설정되었는가 | `<title>`, `<meta description>`, `lang` 확인 |
| 로고·컬러가 `beeliber_design` 기준을 지키는가 | Yellow #FFC700 / Black / White 확인 |
| 콘솔 에러·경고가 없는가 | 브라우저 DevTools Console 확인 |

**보고 한 줄**: `[스봉이] 빌드 ✅ / 금지어 없음 ✅ / SEO ✅ / 콘솔 클린 ✅` 또는 `⚠️ [항목]: [증상]`

---

## 📋 통합 보고 형식

Stitch 작업 완료 후 아래 포맷으로 한 번에 보고합니다:

```
## 🧵 Stitch QA 보고 — [수정된 컴포넌트/화면명]

### 변경 요약
- 무엇을 어떻게 변경했는가 (1~3줄)

### 변경 파일
- client/components/...

### 전 직원 체크 결과
[프엔이] UI 렌더링 ✅ / 반응형 ✅ / 번역 ✅ / 스타일 오염 없음 ✅
[비키]   상태 흐름 ✅ / API 매핑 ✅ / 빈 상태 UI ✅
[예결이] 가격 표시 ✅ / 결제 흐름 ✅ / 권한 노출 없음 ✅
[스봉이] 빌드 ✅ / 금지어 없음 ✅ / SEO ✅ / 콘솔 클린 ✅

### ⚠️ 발견된 이슈 (없으면 "없음")
- [페르소나] 항목: 증상 → 처리 방법

### QA 포인트 (사용자 직접 확인 권장)
- [ ] ...
```

---

## 🚦 이슈 발생 시 처리 원칙

| 심각도 | 기준 | 처리 |
|--------|------|------|
| 🔴 즉시 중단 | 결제 오류 / 데이터 손실 / 보안 노출 | 작업 롤백 후 원인 분석 먼저 |
| 🟡 수정 후 재배포 | 렌더링 깨짐 / 번역 누락 / 콘솔 에러 | 해당 페르소나가 수정 후 재체크 |
| 🟢 다음 이슈로 | 미세 간격 / 색상 미세 조정 | 별도 UI Polish 태스크로 등록 |
