## 1. 두 앤 돈트 (Do and Don't) 대조 지침 [4]
> **담당 에이전트:** `agent_doc_quality` — 문서화 작업, README 작성, 주석 추가 요청 시 호출

* **DO:** 문서화할 때는 코드만 봐서는 알 수 없는 패턴이나 주의 사항만을 작성할 것 [4].
* **DON'T:** 코드를 읽으면 바로 알 수 있는 것은 문서화하지 말 것 [4].
* **길이 제한:** 도구 호출 사이의 텍스트는 25단어 이내로, 최종 응답은 100단어 이내로 유지할 것 [5].

---

## 2. 결과 검증 강제 [5]
> **담당 에이전트:** `agent_verifier` — "완료", "수정", "배포" 선언 직전 자동 호출

* 작업 완료를 보고하기 전에 반드시 테스트를 돌리고 스크립트를 실행하여 출력을 확인하는 등 실제로 작동하는지 검증할 것 [5].
* 실패한 체크를 숨기거나 단순화해서 성공처럼 포장하지 말 것. 결과를 조작하거나 예측하지 말고 현재 상태를 있는 그대로 보고할 것 [2].
* **[중요]** 검증할 수 없는 경우, 추측하지 말고 "검증할 수 없다"고 명시적으로 말할 것 [5].

---

## 3. 턴 예산 및 컨텍스트 비용 관리 (작업 순서) [3]
> **담당 에이전트:** `agent_context_manager` — 다중 파일 수정 작업 시작 전 자동 호출

* 여러 파일을 수정해야 할 경우, 관련된 모든 파일의 읽기(Read) 호출을 병렬로 먼저 발행하여 분석할 것 [3].
* 읽기가 모두 완료된 후에 수정(Write)으로 넘어가며, 읽기와 쓰기를 섞어서 토큰을 낭비하지 말 것 [3, 6].

---

## 4. 체인 오브 소트 스트리핑 (Chain of Thought Stripping) [6, 7]
> **담당 에이전트:** `agent_cot_stripper` — 모든 응답 생성 직전 자동 적용

* 문제 해결을 위한 분석 및 사고 과정은 반드시 `<analysis>` 또는 `<thinking>` 태그 안에 작성할 것 [7].
* 사용자에게 보여지는 최종 출력 결과에는 해당 태그를 제외하고 도출된 결론만 깔끔하게 작성할 것 [7].

---

## 5. 컨텍스트 오염 방지 (작업 경계) [8]
> **담당 에이전트:** `agent_boundary_keeper` — 다중 모듈 작업 및 서브 에이전트 호출 시 자동 호출

* 대화 중간에 포크(Fork) 등의 별도 작업을 시킬 때, 사용자가 명시적으로 요청하지 않는 한 출력 파일을 읽어 맥락을 섞지 말 것 [8].
* 서로 다른 모듈(예: A 모듈, B 모듈)을 다룰 때 각 작업의 에러 로그나 맥락이 유입되지 않도록 작업 경계를 엄격히 유지할 것 [8].

---

## 6. 스킬즈 디스크립션 관리 [9]
> **담당 에이전트:** `agent_skill_manager` — 새 SKILL.md 생성 또는 스킬 목록 20개 초과 시 호출

* 스킬 리스팅에 할당되는 토큰 예산은 전체 컨텍스트 윈도우의 1%에 불과하므로, 커스텀 스킬 디스크립션은 250자 이내로 압축할 것 [9].
* 스킬 개수를 무작정 늘리지 말고 자주 쓰는 핵심 스킬 위주로 관리할 것 [8].
