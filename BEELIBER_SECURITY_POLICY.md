# 🐝 Beeliber 통합 보안 요구사항 문서 (Security Policy)

## 📌 문서 개요

본 문서는 Beeliber의 서울 중심 서비스 확장 및 다중 지점(Multi-Branch) 운영 모델에 최적화된 통합 보안 가이드라인입니다. 기존 RFP의 RBAC 정의와 신규 보안 요구사항을 통합하여 중복을 제거하고, "철통 보안"과 "명품 서비스"를 동시에 구현하는 것을 목적으로 합니다.

---

## 1. 전역 보안 설계 원칙 (Core Principles)

1. **Server-side authorization first**: 프론트 UI 제어는 보조 수단일 뿐, 모든 권한 검증은 서버에서 완결합니다.
2. **Least privilege**: 모든 계정은 업무 수행에 필요한 최소한의 데이터만 접근 가능하도록 합니다.
3. **Object-level authorization**: 주문(Order) 및 증빙 파일(Evidence) 단위로 객체별 접근 제어를 강제합니다.
4. **State machine integrity**: 주문 상태는 정의된 정방향 전이만 허용하며, 모든 변경은 감사 로그를 남깁니다.
5. **Evidence-first operations**: 모든 물품 인계 및 상태 변경 시 사진/로그 기반 증빙을 의무화합니다.
6. **Auditability**: 고위험 액션(환불, 민감정보 조회 등)은 삭제 불가능한 감사 로그 적재가 필수입니다.

---

## 2. 역할 기반 권한 모델 (Role-Based Access Control)

| 역할 (Role)              | 주요 권한 및 열람 범위                                             | 금지 사항                                                      |
| :----------------------- | :----------------------------------------------------------------- | :------------------------------------------------------------- |
| **Super Admin (본사)**   | 전 지점 예약/고객/매출/정산 데이터 무제한 접근, 시스템 설정 제어   | 감사 로그 삭제 불가                                            |
| **Branch Admin (지점)**  | 본인 지점의 예약/고객/정산 내역 조회 및 상태 변경 (접수/보관/인계) | 타 지점 데이터 조회, 총 결제 금액(원가) 열람, 환불 직접 처리   |
| **Partner Storage**      | 할당된 지점의 주문 조회 및 현장 접수/보관 상태 업데이트            | 가격 수정, CSV 다운로드, 운송 상태 제어                        |
| **Carrier Agent (운송)** | 배정된 운송 건의 조회 및 상태 업데이트 (수거/인계/완료)            | 배정되지 않은 주문 조회, 결제 정보 열람, 전체 고객 연락처 접근 |
| **CS Operator**          | 상담 지원 범위 내 주문 조회 및 메모 작성, 취소 요청 접수           | 직접 환불 실행, 가격 수정, 권한 정책 변경                      |
| **Customer**             | 본인의 예약 생성/조회/취소 및 증빙 확인                            | 타인 데이터 접근, 내부 메모/관리자용 로그 열람                 |

## 3. 데이터 보호 및 노출 원칙 (Data Privacy)

### 3-1. 고위험 개인정보 (PII) 관리

- 대상: 이름, 전화번호, 이메일, 예약 일정, 여행 메모, 증빙 사진.
- 마스킹 정책:
  - 전화번호: `010-12**-****`
  - 이메일: `ab***@domain.com`
- 원칙: 검색 결과는 기본 마스킹 처리하며, 상세 페이지 진입 시 개별 접근 권한을 다시 검증합니다.

### 3-2. 주문 상태 머신 (Status Machine)

- **허용 상태**: `CREATED` → `PAID` → `DROP_OFF_PENDING` → `STORED` → `IN_TRANSIT` → `ARRIVED` → `COMPLETED` 등.
- **무결성 보장**: 임의 상태 점프 금지, 낙관적 락(`status_version`)을 통한 동시성 제어.

---

## 4. DB 및 API 보안 가이드라인

### 4-1. PostgreSQL RLS (Row-Level Security) 도입 (P0)

- `orders`, `carrier_assignments`, `evidence_files` 테이블에 RLS를 적용하여 DB 레벨에서 오접근을 차단합니다.
- 예: `USING (source_location_id = ANY(app_allowed_location_ids()))`

### 4-2. API 보안 공통 사항

- 전 엔드포인트 **HTTPS Only** 강제.
- 에러 응답 시 내부 스택트레이스 및 구현 상세 정보 노출 금지.
- 민감 API(정산, 환불, 상태 변경)에 대한 **Rate Limiting** 적용.

---

## 5. 인프라 및 인증 체크리스트

- [ ] **MFA 의무화**: 운영자 및 매니저급 계정은 멀티팩터 인증(MFA) 필수 적용.
- [ ] **보안 헤더**: `CSP`, `HSTS`, `X-Content-Type-Options` 등 필수 헤더 설정.
- [ ] **파일 보안**: 업로드 파일 확장자 제한(jpg, png 등), 저장소 경로 난수화, Signed URL 사용.
- [ ] **감사 로그**: 로그인, 상태 변경, 대량 데이터 다운로드 등 모든 액션에 `actor_id`, `ip`, `timestamp` 기록.
