# 🐝 beeliber 프랜차이즈 ERP 시스템 고도화: 개발 외주 요청서(RFP) 및 기능 명세서

## 1. 프로젝트 개요 (Project Overview)

- **프로젝트명**: beeliber 다중 지점(Multi-Branch) 관리 및 정산 자동화 시스템 구축
- **목적**: 단일 거점 서비스에서 다중 지점(가맹점) 모델로 확장함에 따라, 본사의 완벽한 중앙 통제력 확보 및 지점 간의 독립적인 예약/정산 시스템(B2B SaaS 형태)을 구축합니다.
- **개발 환경**: 현재 운영 중인 안티그래비티(Antigravity) 기반 웹 서비스 고도화

## 2. 사용자 권한 정의 (Role-Based Access Control)

시스템 접근 권한을 철저히 분리하여 데이터 보안과 운영 효율성을 극대화합니다.

| 권한 레벨 | 데이터 열람 범위 | 기능 및 제어 권한 | 블라인드(비노출) 대상 |
| :--- | :--- | :--- | :--- |
| **Super Admin (본사)** | 전체 지점 예약, 전체 고객 정보, 총매출액, 지점별 정산금 | 신규 지점 생성/수정/비활성화, 지점별 수수료율 설정, 전체 예약 상태 강제 변경 | 없음 (모든 데이터 접근 가능) |
| **Branch Admin (지점)** | 본인 지점의 예약 내역, 담당 고객 정보, 본인 지점의 커미션 수익 | 본인 지점의 예약 상태 변경 (픽업/보관/배송완료 처리) | 타 지점 데이터, 고객의 총 결제 금액(원가), 본사 수익 |

## 3. 핵심 기능 명세 (Functional Specifications)

### F-01. [본사 전용] 지점 통합 관리 시스템 (Branch CRUD & Auto-Routing)

본사가 개발자의 개입 없이 새로운 지점을 무한히 확장하고 제어할 수 있는 중앙 관리 기능입니다.

- **신규 지점 등록 (Create)**:
  - 지점 기본 정보 입력 (지점명, 영문 지점 코드(branch_code), 주소, 대표자, 연락처 등).
  - 해당 지점의 관리자 계정(ID/PW) 동시 생성 및 Branch Admin 권한 부여.
  - 해당 지점의 초기 배송/보관 수수료율(%) 기본값 세팅.
- **지점 전용 URL 자동 라우팅 (Dynamic Routing)**:
  - 본사에서 영문 지점 코드(예: yeonnam)를 등록하면, 시스템이 자동으로 서브 디렉토리 형태의 전용 URL(`bee-liber.com/branch/yeonnam`)을 생성하고 해당 지점의 DB와 매핑합니다.
- **지점 정보 수정 및 상태 제어 (Update & Status Control)**:
  - 지점 연락처, 수수료율, 비밀번호 초기화 등 정보 수정 기능.
  - 지점 영업 상태 토글 제어 (운영중 / 임시휴업 / 계약종료).
  - '비활성화(임시휴업/종료)' 처리 시, 고객 프론트 화면(지점 선택 드롭다운 및 전용 URL)에서 즉시 숨김(Hide) 처리되어 예약 유입을 차단합니다.

### F-02. 예약 연동 프론트엔드 (Front-end Integration)

고객이 어떤 경로로 접속하든 정확하게 지점별 DB에 예약이 할당되어야 합니다.

- **본사 메인 페이지 예약**: 예약 플로우 내에 [지점 선택] 드롭다운 UI 추가. 선택된 지점의 `branch_id`가 예약 DB에 함께 저장됨.
- **지점 전용 URL 예약**: (예: `bee-liber.com/branch/yeonnam`) 해당 URL 접속 시, [지점 선택] 단계가 생략되거나 해당 지점으로 고정(Fixed)되어 예약이 진행되며 자동으로 해당 지점 DB로 태깅(Tagging)됨.

### F-03. 커미션(수수료) 자동 정산 엔진 (Settlement Engine)

가장 핵심적인 재무 로직으로, 오차 없는 자동 계산이 필요합니다.

- **서비스별/지점별 개별 수수료율 적용**:
  - 본사는 F-01 메뉴를 통해 지점마다, 그리고 서비스 타입(배송/보관)마다 다른 수수료율(%) 또는 건당 정액(₩)을 설정할 수 있습니다. (예: 연남점 - 보관 60%, 배송 70%)
- **정산 자동화 및 스냅샷 저장 (중요)**:
  - 고객 결제 완료 즉시, 시스템 백단에서 `총 결제 금액 × 지점 수수료율 = 지점 커미션`을 계산합니다.
  - **[DB 설계 필수사항]** 계산된 '최종 지점 정산금'과 당시의 '수수료율'은 해당 주문(Order) 테이블에 **고정값(Hard Copy)**으로 저장되어야 합니다. (이후 본사가 수수료율을 변경해도 과거 정산 내역이 변동되는 치명적 오류를 방지하기 위함입니다.)

### F-04. 재무/통계 대시보드 및 엑셀 다운로드 (Dashboard & Export)

본사와 지점이 각각 필요한 재무 데이터를 확인하고 엑셀로 추출합니다.

- **통계 대시보드 필터링**:
  - 기간 필터 (일간/주간/월간/사용자 지정).
  - 서비스 타입 필터 (전체 / 배송 / 보관).
- **엑셀(Excel) Raw Data 출력 권한 분리**:
  - **Super Admin (본사) 출력 항목**: [예약번호, 결제일시, 지점명, 서비스유형, 고객명, 총결제액, 지점정산액, 본사수익, 진행상태]
  - **Branch Admin (지점) 출력 항목**: [예약번호, 결제일시, 서비스유형, 고객명, 지점정산액, 진행상태] (총 결제액 및 타 지점 정보 절대 노출 차단)
- **환불/취소 로직 동기화**:
  - 예약 취소 또는 부분 환불 발생 시, 해당 건의 '지점 정산액'도 자동으로 마이너스(-) 처리되어 통계 및 엑셀에 실시간 반영되어야 합니다.

## 4. 🚀 개발사 전달 및 협의 사항 (Action Items)

위 기능들을 현재의 안티그래비티 DB 구조(Schema)에 어떻게 추가/변형하여 반영할지 WBS(작업 분할 구조도) 및 산출물 일정을 회신해 주시기 바랍니다.
특히 기존 단일 예약 테이블을 다중 지점 매핑 구조로 마이그레이션(Migration)하는 방안에 대한 기술적 검토를 요청합니다.
